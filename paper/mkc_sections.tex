\subsection{Snowpits and the SnowPilot database}

A snowpit is a trench excavated into the snowpack -- typically to the ground or to a depth
of interest -- that allows an observer to characterize the snow stratigraphy and assess
slope stability. Standard observations include snow layering (layer boundaries, depth, and
thickness), grain characteristics (grain form and grain size), mechanical resistance (hand
hardness index, HHI), density, and temperature profile. Stability tests are often performed
in conjunction with a snowpit; the most common are the Extended Column Test (ECT), the
Compression Test (CT), and the Propagation Saw Test (PST). Terrain characteristics such
as slope angle, aspect, and elevation are also recorded alongside prevailing weather
conditions at the time of observation. Several guiding documents specify standard
observation protocols and recording conventions \citep{fierz2009, greene2022,
caatechnicalcommittee2024, norwegianwaterresourcesandenergydirectorate2022}.

SnowPilot \citep{chabot2004, chabot2016} is a free, open-source web application that
enables practitioners to record, visualize, and store snowpit data in the field. It is
widely used by avalanche professionals and snow scientists, particularly in North America.
Despite its widespread adoption for recording and plotting purposes, the SnowPilot
database has been underutilized as a research resource, due in part to the complexity of
its data format and the absence of purpose-built tools for programmatic analysis
\citep{connelly2025}. This study draws on ten years of SnowPilot data spanning
\mbox{1 September 2014} to \mbox{1 September 2025}; a summary of the dataset is given in
Table~\ref{tab:snowpilot}, and the data processing workflow is described in
Sect.~\ref{sec:data_methods}.

\subsection{Data: SnowPilot and SnowPylot}
\label{sec:snowpylot}

Snowpit data from SnowPilot are stored in the Canadian Avalanche Association Markup
Language (CAAML), a standardized XML schema for snow and avalanche observations
\citep{caatechnicalcommittee2024}. The ten-year dataset used in this study was downloaded
as a collection of individual CAAML XML files, one per snowpit. Each file was parsed into
a structured Python object using SnowPylot \citep{connelly2025}, an open-source Python
library developed for this purpose. Of the files downloaded, one was malformed and could
not be parsed and was therefore excluded, leaving the 51{,}138 snowpits described in
Table~\ref{tab:snowpilot}. Complete documentation of SnowPylot, including the full
parsing specification, is provided in \citet{connelly2025}.

Each parsed snowpit is represented as a \texttt{SnowPit} object comprising three
components. The \texttt{core\_info} component stores pit-level metadata: a unique pit
identifier, recording date, contributing user, and location attributes including slope
angle, aspect, elevation, and country. The \texttt{snow\_profile} component stores the
stratigraphic record as an ordered list of \texttt{layer} objects -- with hand hardness,
primary and secondary grain form, average grain size, and layer thickness -- along with
any temperature measurements at discrete depths and density observations. The
\texttt{stability\_tests} component stores results from stability tests as typed lists:
Extended Column Tests (ECT), Compression Tests (CT), Propagation Saw Tests (PST), and
Rutschblock Tests (RB), each carrying test-specific attributes such as failure score,
failure depth, and fracture character.

Two aspects of the SnowPilot data structure require specific handling in this study.
First, SnowPilot permits density to be entered either as an attribute of a stratigraphic
layer or as a separate density profile composed of discrete depth intervals. Entries in a
stand-alone density profile may not correspond to stratigraphic layer boundaries; only
density measurements explicitly associated with a stratigraphic layer are therefore
retained (Table~\ref{tab:snowpilot}, note~a). Second, SnowPilot records grain form using
the International Classification for Seasonal Snow on the Ground \citep{fierz2009}, which
includes both primary codes (e.g., \texttt{RG} for rounded grains) and compound subform
codes (e.g., \texttt{RGmx} for mixed rounded grains). For parameterization lookup, all
grain form codes are resolved to their primary base code before matching against the
applicability matrices described in Sect.~\ref{sec:parameterizations}.

\subsubsection{Plate theory parameters for composite slabs}

% ============================================================
% 2.3.5 PLATE THEORY PARAMETERS
% Classical laminate theory following Weißgraeber and Rosendahl (2023)
% ============================================================
% - Snow slab modeled as a multi-layer plate; each layer treated as
%   homogeneous, isotropic, linearly elastic, perfectly bonded
% - Define reduced stiffness of each layer: Q_i = E_i / (1 - nu_i²)
% - A11 (extensional stiffness, N/mm): A11 = Σ Q_i * h_i
% - B11 (bending-extension coupling, N): B11 = (1/2) * Σ Q_i * (z_{i+1}² - z_i²)
% - D11 (bending stiffness, N·mm): D11 = (1/3) * Σ Q_i * (z_{i+1}³ - z_i³)
%   where z_i is the position of the bottom face of layer i measured from
%   the bottom of the slab
% - A55 (transverse shear stiffness, N/mm): A55 = κ * Σ G_i * h_i, with κ = 5/6
% - Static stress applied to weak layer by overlying slab:
%     sigma_zz = Σ rho_i * g * h_i * cos(theta)  (normal)
%     sigma_xz = Σ rho_i * g * h_i * sin(theta)  (shear)
%   (equations already in working_draft.tex)
% - Note that D11 is particularly important: stiffer, thicker slabs store
%   more elastic energy and thus drive crack propagation more effectively

\subsubsection{Representing parameterizations as a directed graph}

% ============================================================
% 2.4.1 THE PARAMETERIZATION GRAPH
% ============================================================
% - Some parameters are inputs to other parameterizations (method chaining),
%   creating complex dependency relationships (density → E → D11)
% - Represent parameterizations as a directed acyclic graph (DAG):
%     * Parameter nodes: observed quantities (hand hardness, grain form, etc.)
%       and derived quantities (density, E, D11, etc.)
%     * Method nodes (merge nodes): combine required inputs to compute output;
%       implement AND logic (all inputs required)
%     * Edges: data flow from parameter to method to output parameter
%     * OR logic at parameter nodes: multiple incoming methods = multiple pathways
% - Present Figure: full parameterization graph (params_graph figure)
%   with node types, levels (layer vs. slab), and edge labels
% - Node levels: layer-level (density, E, nu, G) and slab-level (A11, B11, D11, A55)
% - The shared density node is a key structural feature: density feeds both
%   E-mod and Poisson's ratio (Srivastava), so the same density method must be
%   used consistently within a pathway

% ============================================================
